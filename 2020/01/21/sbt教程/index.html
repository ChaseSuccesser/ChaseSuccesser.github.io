# 自定义SBT下载的jar包存放的路径

-Dsbt.boot.directory=e:/sbt/boot/

-Dsbt.ivy.home=e:/sbt/

打开命令行窗口，执行命令sbt

这样就会开始下载相关的jar包，同时观察设置的sbt.boot.directory及sbt.ivy.home

查看SBT版本：

进入SBT命令模式，键入sbtVersion，就可以查看SBT的版本了。

SBT项目的工程结构：

SBT的构建文件

1、build.sbt介绍

name := "hello" // 项目名称 (require)

organization := "xxx.xxx.xxx" // 组织名称

version := "0.0.1-SNAPSHOT" // 项目版本号 (require)

scalaVersion := "2.9.2" // 使用的Scala版本号

// name和version的定义是必须的 ，因为如果想生成jar包的话，这两个属性的值将作为jar包名称的一部分。

  • name如果没有设置，将会是默认值"default"；
  • rganization如果没有设置，也将会是默认值"default"；
  • version如果没有设置，默认值是"0.1-SNAPSHOT"；

SBT的name和organization等价于Maven中的artifactId和groupId；

在build.sbt中添加配置需要遵循一些语法限制：

  1. Any configuration setting/definition (values or methods) must be declared without any blank lines.

  2. There must be at least one blank line between each configuration setting/definition.

2、project目录下的几个文件介绍

project目录下的几个文件实际上都是非必须存在的，可以根据情况添加。

build.properties ：声明要使用哪个版本的SBT来编译当前项目。

最新的sbt boot launcher可以能够兼容编译所有0.10.x版本的SBT构建项目，比如如果我使用的是0.12版本的sbt，但却想用0.11.3版本的sbt来编译当前项目，则可以在build.properties文件中添加sbt.version=0.11.3来指定。

默认情况下，当前项目的构建采用使用的sbt boot launcher对应的版本。

plugins.sbt ：用来声明当前项目希望使用哪些插件来增强当前项目使用的sbt的功能。

比如像assembly功能，清理ivy local cache功能，都有相应的sbt插件供使用， 要使用这些插件只需要在plugins.sbt中声明即可

3、.scala形式的build定义

如果需要 自定义Task 、 多模块的项目构建 ，就必须使用.scala形式的build定义.

.sbt能干的事情，.scala形式的build定义都能干；反之，则不然。

Build.scala的一个demo：

importsbt._importKeys._

objectHelloBuildextendsBuild {

overridelazyvalsettings=super.settings ++Seq(..)

lazyvalroot=Project(id =&quot;hello&quot;,

                        base = file(&quot;.&quot;),

                        settings =Project.defaultSettings ++Seq(..))

}

.sbt和.scala之间不得不说的那些事儿：

  1. 两种形式可以共存，我们可以在.sbt中定义常用的settings，而在.scala中定义自定义的其它内容。

    SBT在编译期间，会将.sbt中的settings的定义与.scala中的定义合并，作为最终的build定义使用。

  2. .sbt和.scala二者之间的settings是可互相访问的， .scala中的内容会被import到.sbt中，而.sbt中的settings也会被添加到.scala的settings当中。

默认情况下，.sbt中的settings会被纳入Project级别的Scope中，除非明确指定哪些Settings定义的Scope；.scala中则可以将settings纳入Build级别的Scope，也可以纳入Project级别的Scope。

4、SBT项目结构的本质 【重要】

首先， 一个SBT项目，与构建相关联的基本设施可以概况为3个部分， 即：

  • 项目的根目录，比如hello/，用来界定项目构建的边界；
  • 项目根目录下的*.sbt文件，比如hello/build.sbt，用来指定一般性的build定义；
  • 项目根目录下的project/*.scala文件，比如hello/project/Build.scala，用来指定一些复杂的，*.sbt形式的build定义文件不太好搞的设置；

也就是说， 对于一个SBT项目来说，SBT在构建的时候，只关心两点：

  • build文件的类型（是*.sbt还是*.scala）；
  • build文件的存放位置（*.sbt文件只有存放在项目的根目录下， SBT才会关注它或者它们；*.scala文件只有存放在项目根目录下的project目录下，SBT才不会无视它或者它们。实际上只有那些定义了扩展自sbt.Build类的scala文件，才会被认为是build定义）

在以上基础规则的约束下，我们来引入一个推导条件， 即：

  • 项目根目录下的project目录，其本身也是一个标准的SBT项目。
  • project目录下的 .scala 文件
    • .scala文件既是上层目录界定的SBT项目的build定义文件，也是上层目录界定的SBT项目的源代码，
    • .scala文件也是project这个目录下的SBT项目的源代码。

如果把project目录下的.scala文件当作是project这个目录下的SBT项目的源代码来看待，因为它是一个源文件，如果它也需要第三方依赖的话，就需要在project目录下新建build.sbt文件，将依赖添加进去。

因为project目录也是一个SBT项目，如果它的构建也要用到自定义task或者command等，该怎么办呢？

前面讲了像自定义task等这样任务必须使用.scala形式的build定义；而且前面又讲了project目录下的.scala文件是上层目录界定的SBT项目的build定义文件。

所以，对于刚刚说的这种情况，可以在project目录下再新建一个project目录，在新建的project目录中再添加.scala文件作为上层project这个SBT项目的build定义文件。

也就是说，从第一层的项目根目录开始， 其下project目录内部再嵌套project目录，可以无限递归，而且每一层的project目录都界定了一个SBT项目，而每一个下层的project目录界定的SBT项目其实都是对上一层的SBT项目做支持，作为上一层SBT项目的build定义项目。

SBT命令

clean 删除所有生成的文件 （在 target 目录下）。

compile 编译源文件（在 src/main/scala 和 src/main/java 目录下）。

test 编译和运行所有测试。

package 将 src/main/resources 下的文件和 src/main/scala 以及 src/main/java 中编译出来的 class 文件打包成一个 jar 文件。

console 进入到一个包含所有编译的文件和所有依赖的classpath的Scala解析器。输入:quit，Ctrl+D（Unix），或者Ctrl+Z（Windows）返回到sbt。

run &lt;参数&gt;* 在和 sbt 所处的同一个虚拟机上执行项目的 main class。

help &lt;命令&gt; 显示指定的命令的详细帮助信息。如果没有指定命令，会显示所有命令的简介。

reload重新加载构建定义（build.sbt，project/*.scala，project/*.sbt这些文件中定义的内容)。在修改了构建定义文件之后需要重新加载。

上面的命令在某些情况下也可以结合SBT的触发执行机制一起使用， 唯一需要做的就只是在相应的命令前追加~符号

$ sbt ~compile

上面命令意味着， 我更改了任何源代码并且保存之后，将直接触发SBT编译相应的源代码以及相应的依赖变更。

SBT的核心概念 【重要】

SBT的核心东西就是Setting、Task、Configuration、Project。

Using just these concepts, you can construct any build for any project.

一、Setting

在SBT中，一个setting就是一个值 。它可能是项目工程的名字，或者是Scala的版本号，等等。

一个setting包括了三个部分：一个key，一个操作符，一个initialization。

在SBT中，每一个key有且只有一种类型，任何赋予key的值必须符合对应的类型，否则build fail。

settings are executed when the build is first loaded.

在SBT中，有三种方式用来创建setting：

  • := 给一个key分配一个initialization expression。会覆盖旧的值。
  • += 在一个key中，追加一个initialization expression。
  • ++= 在一个key中，追加一个initialization expression sequence。

Initializations

initialization是一个Scala表达式，用来产生setting的值。

initialization是Scala代码，因此可以读取Environment variables、properties、file、or anything else available in the JVM。

并且，通过settingKey .value 方法，它可以使用其它setting的值。

Initialization的作用： 在一个地方定义好了一个setting，在其它地方可以重用。而不需要再重新定义。

二、Task

Task用来执行重复的操作 ，例如编译工程、生成文档、运行Test等等。

  • SBT不像Maven，它没有phases、goals、executions。如果你想做某些事情，只需要执行一个task。
  • 如果你希望一个task在另一个task执行完之后再执行，只需要显示地添加两者的依赖关系即可。

默认情况下，所有的tasks是并行执行的。但是可以通过dependency tree，显示指定哪些是串行执行，哪些是并行执行。

通过并行执行地执行task，可以加速build的速度。

  • 在SBT中，task就是Scala代码。可以直接在build 配置中写Scala代码来创建task。

1. 创建Task和使用Task的步骤

(1). 定义Task

Task的定义分为两部分：第一部分就是要定义一个TaskKey来标志Task， 第二部分则是定义Task的执行逻辑。

定义一个TaskKey示例：

val hello = TaskKey[Unit]("hello", "just say hello")

① 因为task的执行结果只是打印字符串，所以指定为Unit类型；

② 第一个参数为TaskKey的名称；

③ 第二个参数为TaskKey的描述；

④ hello是一个key的名称，用来存储task的结果。

通过 := 方法** 将相应的key和执行逻辑定义关联到一起 **：

hello := {

println(&quot;hello, sbt~&quot;)

}

(2). 将task添加到项目的settings当中

object ProjectBuild extends Build {

val hello = TaskKey[Unit]("hello", "just say hello")

val helloTaskSetting = hello := {

println(&quot;hello, sbt~&quot;)

}

lazy val root = Project(id = "", base = file(".")).settings(Defaults.defaultSettings ++ Seq(helloTaskSetting): _*)

}

(3). 使用自定义的Task

$ sbt hello

hello, sbt~

// 或者

$ sbt

&gt; hello

hello, sbt~

注意：

当一个Task执行失败了，不会影响其它Task的执行。

2、Task dependencies

可以在Task的执行逻辑中，通过使用taskKey .value 方法来使用其它Task的执行结果。

但是，在SBT的0.13版本之前，如果要在一个task或setting中使用其他task或setting的值，需要像下面的方式来实现：

zipPath &lt;&lt;= target map {

(t: File) =\&gt; t / &quot;out.zip&quot;

}

而在0.13版本，SBT使用Scala macro简化了这一操作：

zipPath := target.value / "out.zip"

注意：

只能在task或setting中才能通过value方法来访问其他task或setting的结果，如果在method中这样做，将会报错。

要想在method中使用其他task或setting的结果，需要把结果值当做method的参数传给method。

总结Setting和Task：

两者都可以产生值，并且Setting或Task内部也可以使用其它Setting或Task产生的值。

三、Configurations

Configuration被用来隔离多个Tasks或者多个Settings。相当于为Settings或Tasks分配了命名空间。

SBT内置的configurations：

  • Compile：These settings and values are used to compile the main project and generate production artifacts.
  • Test ：These settings and values are used to compile and run unit-testing code.
  • Runtime：These settings and values are used to run your project within sbt.
  • IntegrationTest：These settings and values are used to run tests against yourproduction artifacts

除了上面的configuration，SBT还提供了 Default, Pom, Optional, System, Provided, Docs, and Sources等configuration。

四、多模块工程管理(Multi-Module Project)

包含多个子模块或者子项目的SBT项目跟一个标准的独立的SBT项目相差不多，唯一的差别在于：

  1. build定义中多了对多个子模块/工程的关系的描述；

  2. 项目的根目录下多了多个子模块/工程相应的目录；

多模块工程的典型结构：

${project.root}

** - build.sbt**

+ src/main/scala

** + project**

** - Build.scala**

+ module1

** - build.sbt**

    + src/main/scala

+ module2

** - build.sbt**

    + src/main/scala

+ module3

** - build.sbt**

+ src/main/scala

注意：每个子模块/项目下不可以再创建project目录下其下相应的*.scala定义文件（实际上可以创建，但会被SBT忽略）

子模块/工程与当前项目或者其它子模块/工程之间的关系只能在 *.scala形式的build定义 中进行声明。

例如在${project.root}/project/Build.scala)中我们可以简单的定义多个子模块/工程之间的关系如下：

import sbt._

import Keys._

object MultipleModuleProjectBuild extends Build {

lazy val root = Project(id = &quot;root&quot;, base = file(&quot;.&quot;)) **aggregate(sub1, sub2)**

lazy val sub1 = Project(id = &quot;m1&quot;, base = file(&quot;module1&quot;))

lazy val sub2 = Project(id = &quot;m2&quot;, base = file(&quot;module2&quot;)) **dependsOn(sub3)**   //m2依赖m3

lazy val sub3 = Project(id = &quot;m3&quot;, base = file(&quot;module3&quot;))

}

说明：

  1. aggregate 指明的是一种并行的相互独立的行为，只是这种行为是随父项目执行相应动作而触发的， 比如，在父项目中执行compile，则会同时触发module1和module2两个子模块的编译，两个子模块之间的执行顺序等行为并无联系

  2. dependsOn 则说明一种强依赖关系，像module2这个子项目，因为它依赖module3，所以，编译module2子模块/项目的话，会首先编译module3，然后才是module2; 当然，module3的相应artifact也会加入到module2的classpath中。

在Maven的多子模块/项目的组织结构中，我们很喜欢将所有项目可以重用或者共享的一些设定或者依赖放到父项目的build定义中，在SBT中也是可以的，只要在父项目的.sbt或者.scala形式的build定义中添加即可，不过，要将这些共享的设定的scope设定为 ThisBuild ， 例如：

scalaVersion in ThisBuild := "2.9.2"

这样，就不用在每一个项目/模块的build定义中逐一声明要使用的scala版本了。

使用projects命令查看当前工程中有哪些子模块。

如果想让SBT运行某个子模块的task，可以按照下面的方式来做：

子模块名 / task

如果没有子模块名称的前缀，将会在所有的子模块中执行task。

Project对象有个settings方法，可以在方法里添加一些setting。例如：

lazy val analytics = (

Project("analytics", file("analytics"))

dependsOn(common)

settings (

version := "1.0",

organization := "com.preownedkittens",

libraryDependencies += "org.specs2" % "specs2_2.10" % "1.14" % "test"

)

)

SBT Launcher Configuration 【重要】

文档地址：http://www.scala-sbt.org/0.13/docs/Launcher-Configuration.html

The default configuration file for sbt as an application looks like:

[scala]

version: ${sbt.scala.version-auto}

[app]

org: ${sbt.organization-org.scala-sbt}

name: sbt

version: ${sbt.version-read(sbt.version)[0.13.5]}

class: ${sbt.main.class-sbt.xMain}

components: xsbti,extra

cross-versioned: ${sbt.cross.versioned-false}

[repositories]

local

typesafe-ivy-releases: http://repo.typesafe.com/typesafe/ivy-releases/, [organization]/[module]/[revision]/[type]s/artifact.[ext], bootOnly

maven-central

sonatype-snapshots: https://oss.sonatype.org/content/repositories/snapshots

[boot]

[ivy]

repository-config: ${sbt.repository.config-${sbt.global.base-${user.home}/.sbt}/repositories}

SBT Launcher Configuration的详细说明：

1、[scala]部分被用来配置Scala的版本。

它包含如下几个属性：

①version - The version of Scala an application uses, or auto if the application is not cross-versioned.

②classifiers - The (optional) list of additional Scala artifacts to resolve, e.g. sources.

2、[app]部分用来配置如何让Launcher使用ivy dependency manager去寻找你的application。

它包含如下属性：

①org - The organization associated with the Ivy module. (groupId in Maven vernacular)

②name - The name of the Ivy module. (artifactId in Maven vernacular)

③version - The revision of the Ivy module.

④class - The name of the "entry point" into the application.

An entry point must be a class which meets one of the following criteria(条件)：

     Extends the xsbti.AppMain interface.

Extends the xsbti.ServerMain interfaces.

Contains a method with the signature static void main(String[])

Contains a method with the signature static int main(String[])

Contains a method with the signature static xsbti.Exit main(String[])

⑤components - An optional list of additional components that Ivy should resolve.

⑥cross-versioned - An optional string denoting(指示) how this application is published.

⑦resources - An optional list of jar files that should be added to the application's classpath.

⑧classifiers - An optional list of additional classifiers that should be resolved with this application, e.g. sources.

3、[repositories] section configures where and how Ivy will look for your application.

Each line denotes a repository where Ivy will look.

There are several built-in strings that can be used for common repositories：

①local - the local Ivy repository ~/.ivy2/local.

②maven-local - The local Maven repository ~/.m2/repository.

③maven-central - The Maven Central repository repo.maven.org.

Besides built in repositories, other repositories can be configured using the following syntax：

** ** name: url(,pattern)(,descriptorOptional)(,skipConsistencyCheck)

各部分说明：

①name property is an identifier which Ivy uses to cache modules resolved from this location.

②url property is the base url where Ivy should look for modules.

③pattern property is an optional specification of how Ivy should look for modules.

By default, the launcher assumes repositories are in the maven style format.

④skipConsistencyCheck string is used to tell Ivy not to validate checksums and signatures of files it resolves.

4、[boot] section is used to configure where the sbt launcher will store its cache and configuration information.

它包含如下的属性：

①directory - The directory is used to store all cached JARs resolved launcher.

②properties - (optional) A properties file to use for any read variables.

5、[ivy] section is used to configure the Ivy dependency manager for resolving applications.

它包含如下的属性：

① ivy-home - The home directory for Ivy.

This determines where the ivy-local repository is located, and also where the Ivy cache is stored.

Defaults to ~/.ivy2

②checksums - The comma-separated(逗号分隔) list of checksums that Ivy should use to verify artifacts have correctly resolved,

e.g.md5 or sha1.

③ repository-config - This specifies a configuration location where Ivy repositories can also be configured.

If this file exists, its contents will override the [repositories] section.

Variable Substitution

Property values may include variable substitutions. A variable substitution has one of these forms：

${variable.name}

${variable.name-default}

说明：

variable.name is the name of a system property. If a system property by that name exists, the value is substituted.

If it does not exists and a default is specified, the default is substituted after recursively substituting variables in it.

If the system property does not exist and no default is specified, the original string is not substituted.

在终端使用SBT命令时，让SBT使用我们指定的目录：

修改conf目录下的sbtconfig.txt文件：

-Dsbt.ivy.home=E:\ivy-repository

-Dsbt.boot.directory=E:\ivy-repository\boot

(

-sbt-dir ~/Work/sbt

-sbt-boot ~/Work/sbt/boot

-ivy ~/Work/ivy-repository

)

做完上面的配置后，在终端使用sbt相关命令的时候，都会使用指定的目录。

不过，实际开发中需要依赖IDEA，下面介绍如何在IDEA中也使用指定的目录。

IDEA中对SBT的配置：

  1. 使用custom的sbt-launch.jar。

  2. 由于IDEA是通过sbt-launch.jar来操作sbt的，并不是sbt命令。所以前面对sbtopts文件做的修改对IDEA不管用，还是会在用户目

    录下自动创建.sbt和.ivy2文件夹。

    如果把sbt-launch.jar解压出来，会看到在sbt目录下有个sbt.boot.properties文件（这个文件里内容以及作用前面已经介绍了）。

    我们需要做的就是：新建一个repositories文件，并添加sbt.repository.config系统属性，覆盖默认的[repositories]配置。

(1) 添加系统属性

在Build Tools下的sbt设置,有一个JVM Options,可以设置堆大小,和其他的jvm参数(VM parameters),在这里设置就行了

-XX:MaxPermSize=384M // 这是已有的默认值

-Dsbt.ivy.home=E:\ivy-repository // 新加

-Dsbt.boot.directory=E:\ivy-repository\boot // 新加

-Dsbt.repository.config=/Users/lgx/.sbt/ // 新加

(2) 新建的repositories文件的内容

[repositories]

local

oschina nexus:http://maven.oschina.net/content/groups/public/

oschina-ivy:http://maven.oschina.net/content/groups/public/, [organization]/[module]/(scala_[scalaVersion]/)(sbt_[sbtVersion]/)[revision]/[type]s/artifact.[ext]

依赖 【重要】

一、SBT依赖管理

SBT的managed dependencies采用 Apache Ivy 的依赖管理方式， 可以支持从Maven或者Ivy的Repository中自动下载相应的依赖。

简单来说，在SBT中， 使用managed dependencies基本上就意味着往 libraryDependencies 这个Key中添加所需要的依赖.

在build.sbt文件中，添加的一般格式：libraryDependencies += groupID % artifactID % revision

  • libraryDependencies变量 的类型是Seq[ModuleID]。
  • % 方法 被用来创建ModuleID对象。

ModuleID包含三个必须的部分：organization、name、version。这是Maven的groupId、artifactId、version的变种。

其他格式:

// 限定依赖的范围只限于测试期间

libraryDependencies += "org.apache.derby" % "derby" % "10.4.1.3" % "test"

// 排除递归依赖中某些我们需要排除的依赖。如果要排除多个依赖，就在后面继续调用exclude方法即可。

libraryDependencies += "org.apache.derby" % "derby" % "10.4.1.3" exclude("org", "artifact")

libraryDependencies += "org.apache.derby" %% "derby" % "10.4.1.3"

// 在查找依赖时，将当前项目使用的scala版本号追加到artifactId之后作为完整的artifactId来查找依赖

比如如果我们的项目使用scala2.9.2，那么(3)的依赖声明实际上等同于"org.apache.derby" % "derby_2.9.2" % "10.4.1.3"，这种方式更多是为了简化同一依赖类库存在有多个Scala版本对应的发布的情况。

除了使用 += 方法，还可以使用 ++= 方法来添加依赖：

libraryDependencies ++= Seq(

"junit" % "junit" % "4.11" % "test",

"org.specs2" % "specs2_2.10" % "1.10" % "test"

)

二、Resovers简介

对于managed dependencies来说, 默认情况下，SBT会去默认的Maven的Repository中抓取依赖，

如果默认的Repository中找不到我们的依赖，那我们可以通过resolver机制，追加更多的repository让SBT去查找并抓取.

例如:

resolvers += "Sonatype OSS Snapshots" at "https://oss.sonatype.org/content/repositories/snapshots"

注意：at之前是要追加的repository的标志名称（任意取），at后面则是要追加的repository的路径。

除了可远程访问的Maven Repo，我们也可以将本地的Maven Repo追加到resolver的搜索范围：

resolvers += "Local Maven Repository" at"file://"+Path.userHome.absolutePath+"/.m2/repository"

三、解决依赖冲突

一般情况下，SBT会在target/resolution-cache/reports/目录下dump出项目依赖的reports。

每一个report是一个XML，并且包含格式化的信息，所以可以在Web浏览器里查看这些reports。（浏览器须是火狐，谷歌不能查看本地机器的XSTL，所以会有问题）

为了让SBT使用proxy repository，需要配置两个地方：~/.sbt/repositories文件、以及launcher script。

官方文档地址：http://www.scala-sbt.org/0.13/docs/Proxy-Repositories.html

1、~/.sbt/repositories

repositories文件是SBT Launcher的额外配置。这个配置文件的详细语法参考：【SBT Launcher Configuration】

打包

SBT打包后的jar包可以deploy到lvy或者maven仓库中。

  • SBT的打包任务是交给 package task来执行。
  • package task依赖packageBin task。packageBin task用来创建JAR包。
  • JAR包里包含什么文件是通过mappings in packageBin task来定义的。这个task的类型是Seq[(File, String)]，第一个元素表示包含在jar中的文件，第二个元素表示文件在jar中的存储路径。

在SBT中，classes的目录并不是直接就在target目录下面。而是类似这样：target/scala-2.10/classes 和 target/scala-2.10/test-classes。不像Maven是这样的：target/classes。

SBT这样做的原因是可以使用多个Scala版本进行编译。

如果想使用多个Scala版本编译代码，可以添加下面的配置：

crossScalaVersions := Seq("2.8.2", "2.9.2", "2.10.0")

然后执行"+package" task，SBT将会使用你指定的每一个Scala版本去编译代码，并产生3个JAR包。

Standard organization of sources

&gt; show javaSource // 显示Java源码的路径

&gt; show scalaSource // 显示Scala源码的路径

&gt; show resourceDirectory // 显示资源文件所在的路径

发布包

  • publish：发布项目的artifact到一个远程的仓库，例如Maven或lvy。
  • publishLocal：发布项目的artifact到本地机器的lvy仓库中（&lt;HOME&gt;/.ivy2/local/）。
  • publishM2：发布项目的artifact到本地机器的Maven仓库中。

Publish remotely

在build.sbt文件中为你的项目配置发布路径：

publishTo := Some("preowned-kittens-release" at "http://prownedkittens.artifactoryonline.com/preowned-kittens/mvn-releases")

默认情况下，SBT有一个针对全局配置的文件：位于~/.sbt/0.13/目录下的以.sbt后缀的文件。这些文件中的配置信息会作用到本地机器上所有以SBT为构建工具的项目。

因为要发布到Maven仓库中，还需要做另外一件事，就是配置Maven的认证信息(用户名/密码)。

对于我们Maven认证信息的配置，可以在~/.sbt/0.13/目录下创建一个user.sbt文件，文件内容有：

credentials += Credentials("Artifactory Realm", "preownedkittens.artifactoryonline.com", "joe.developer", "my-encryptedpassphrase")

credentials由四部分组成：

  • The realm—The challenge string issued by the HTTP server when it asks fora password
  • The host—The hostname of the server the credentials are for
  • The username—Your Artifactory Online username
  • The passphrase—Your Artifactory Online passphrase

SBT Plugin

A plugin is a jar that contains settings or tasks.

The jar contains a definition file that sbt reads to work out what settings and tasks are available.

SBT Plugin机制允许我们扩展SBT项目的build定义。这里的扩展基本可以理解为允许我们向项目的build定义里添加各种所需的Settings，比如自定义Task。

既然Plugin是对build定义的扩展，那么我们就可以认为项目的build定义依赖这些plugin的某些状态或者行为，即plugin属于项目build定义的某种依赖。从这个层次来看，plugin的配置和使用跟library dependency的配置和使用是一样的(具体有稍微的差异)。

既然plugin是对build定义的扩展（及被依赖），那么，我们应该在build定义对应的SBT项目的build定义中配置它，即 hello/project/*.sbt 或者 hello/project/project/*.scala 。大多数情况下，我们会直接在像hello/project/plugins.sbt文件中配置和添加Plugin。

(plugins.sbt的名称只是便于识别，实际上，SBT只关注.sbt的后缀，具体名称是不关心的.

plugins.sbt本质上也是一个SBT项目的build定义文件，除了在其中配置Plugin，我们同样可以添加第三方依赖， 追加其他Setting等，跟一般的.sbt配置文件无异

)

示例：

resolvers += Resolver.url("git://github.com/jrudolph/sbt-dependency-graph.git")

resolvers += "sbt-idea-repo" at "http://mpeltonen.github.com/maven/"

addSbtPlugin("com.github.mpeltonen" % "sbt-idea" % "1.1.0")

addSbtPlugin("net.virtual-void" % "sbt-dependency-graph" % "0.6.0")

//如果Plugin是发布到SBT默认会查找的Maven或者Ivy Repository中，则只需要addSbtPlugin就行了， 否则， 需要将Plugin发布到的Repository添加到resolvers以便SBT可以发现并加载成功。

SBT解析plugin的方式和解析dependency的方式有点差别：

addSbtPlugin("org.scalastyle" %% "scalastyle-sbt-plugin" % "0.5.0")

SBT会把上面Plugin的配置转换为URL：(**含有Scala版本、SBT版本、Plugin版本**)

http://repo1.maven.org/maven2/org/scalastyle/scalastyle-sbt-plugin\_2.10\_0.13/0.5.0/scalastyle-sbt-plugin-0.5.0.jar

注意：

在URL中添加的Scala版本号并不是在build.sbt中声明的Scala版本号，而是SBT使用的Scala版本号。

每一个版本的SBT都是使用指定的Scala版本进行编译的：0.13版本的SBT使用的是2.10版本的Scala，0.12.x版本的SBT使用的是2.9版本的Scala。

libraryDependencies += "org.specs2" %% "specs2" % "1.14"

SBT会把上面dependency的配置转换为URL：(**含有Scala版本，Dependency版本，没有SBT版本**)

https://repo1.maven.org/maven2/org/specs2/specs2\_2.10/1.14/specs2\_2.10-1.14.jar

常用的Plugin:

1. sbt-idea

sbt-idea这个插件可以帮助我从SBT的配置中生成IDEA这个IDE对应的classpath，项目信息等元数据， 只要运行sbt gen-idea这一命令之后，就可以在IDEA中直接打开当前的SBT项目了。

2. sbt-dependency-graph

项目的依赖越多，依赖关系就越复杂，这个插件可以帮助我们理清楚项目各个依赖之间的关系，完全跟Maven的dependency:tree类似的功能.

3. sbt-assembly

可以将当前项目的二进制包以及依赖的所有第三方库都打包成一个jar包发布，即one-jar，对于那种直接运行的应用程序很方便。

在project/plugins.sbt中添加：

addSbtPlugin("com.eed3si9n" % "sbt-assembly" % "0.11.2")

然后，在build.sbt添加下面的配置：

import AssemblyKeys._

assemblySettings

然后，就可以使用了：

&gt; assembly

如果上面的assembly命令执行结果显示有重复的文件冲突，可以在build.sbt添加下面的配置来解决：

// 定义如何处理重复的文件

mergeStrategy in assembly &lt;&lt;= (mergeStrategy in assembly) { (old) =&gt;

{

    case &quot;application.conf&quot; =\&gt; MergeStrategy.concat

    case &quot;reference.conf&quot; =\&gt; MergeStrategy.concat

    case &quot;META-INF/spring.tooling&quot; =\&gt; MergeStrategy.concat

    case &quot;overview.html&quot; =\&gt; MergeStrategy.rename

    case x =\&gt; old(x)

}

}

// 定义在打jar包时，哪些依赖jar应该被排除

excludedJars in assembly &lt;&lt;= (fullClasspath in assembly) map { cp =&gt;

cp filter { f =\&gt;

    (f.data.getName contains &quot;commons-logging&quot;) ||

    (f.data.getName contains &quot;sbt-link&quot;)

}

}

为了使用java -jar命令来执行打包后的jar，需要指定一个main类：

mainClass in assembly := Some("Global")

加了上面这行的配置后，assembly插件会帮助创建MANIFEST.MF文件。

上面的配置做完之后，就可以重新执行assembly命令打jar包了。

打完jar包之后，就可以使用 java -jar xxx.jar 来启动你的应用程序了。

4. sbt-dirty-money

SBT会将项目的依赖抓取到本地的ivy cache中缓存起来，避免频繁的update对带宽造成的浪费，但有些时候需要对缓存里失效的内容进行清理，使用这个插件可以避免自己手动遍历目录逐一删除相应的artifacts

5. sbt-revolver

使用这个插件可以自动地重启应用，只要它能发现一个含有main方法的类。

在project/plugins.sbt文件添加：

addSbtPlugin("io.spray" % "sbt-revolver" % "0.7.2")

然后，在build.sbt中添加下面的settings：

Revolver.settings

然后，就可以使用了：

&gt; reStart #build project and start(or restart) project

&gt; reStop #stop the project if it`s running

&gt; ~reStart #restart project when source files change

process

在Process()方法中我们可以执行命令。例如shell命令，java命令等等。

在Process()方法后面添加感叹号!，表示抛弃命令的执行结果，不关心这些内容。

可以使用#&gt;或者#&gt;&gt;，把process的输出重定向到其他地方。

例如：

url("http://databinder.net/dispatch/About") #&gt; "grep JSON" #&gt;&gt; file("About_JSON") !

读取一个url链接的内容，把获取到的内容交给grep命令做过滤，然后将过滤的内容追加到文件中。

组合多个Process：

a #&& b

如果a执行成功(zero exit code)，则执行b，最后返回b的结果状态码；

如果a执行失败(non-zero exit code)，不执行b，最后返回a的结果状态码。

a #|| b

如果a执行成功，不执行b，最后返回a的结果状态码；

如果a执行失败，则执行b，最后返回b的结果状态码；

a #| b

a和b都会执行，会把a的执行结果通过管道输送给b。

log

SBT提供了一个标准的日志框架： Streams 。可以使用它来追踪我们的build输出。

Streams有几种日志级别：error、warn、info、debug。

在一个task中使用Streams是很简单的，只需要使用streams.value.log就可以得到一个Streams的实例。

示例：

val log = streams.value.log

log.debug("value=" + content)

SBT会把日志内容输出到文件和屏幕(依赖日志级别)

  • 默认情况下，debug级别的日志只会输出到文件中。
  • 默认情况下，SBT只会在控制台显示error、warn级别的日志。

不过我们可以以task、configuraation、project、globally维度来控制日志的级别。

以task的维度设置日志级别：set logLevel in &lt;taskKey&gt; := Level.Debug

以globally的维度设置日志级别：set logLevel in Global := Level.Warn

可以使用"last"命令查看日志内容：

&gt; last

还可以在last后面追加一个先前执行过的命令，这可以查看先前命令执行的日志：

&gt; last compile

需要注意：

我们不能直接在method中使用streams.value.log，

如果在一个task中使用了我们自定义的方法，可以通过在task中创建Streams实例，然后传递给方法。

测试

一、SBT与JUnit

添加依赖：

libraryDependencies += "junit" % "junit" % "4.11" % "test"

libraryDependencies += "com.novocode" % "junit-interface" % "0.11" % "test" ** //加上此依赖，SBT才会执行JUnit test class**

  • 如果只添加上面第一个依赖，在执行test task时，并不会运行我们写的JUnit test。
  • 这是因为SBT提供给了一个test-interface，SBT通过它来找到我们定义的test class，并选用指定的test framework来运行对应的test class。
  • SBT默认支持ScalaTest、ScalaCheck、specs2，因为这些test framework内部实现了test-interface。

但是JUnit并没有实现test-interface，因此SBT默认不会执行JUnit test class。

使用SBT的技巧

1. 在SBT中如何mvn dependency:tree

答案：使用sbt-dependency-graph插件！

因为它不是项目的功能依赖， 而它又是很通用的功能，所以， 最合适的做法就是把它配置成global的plugins。

在SBT 0.13.x中，~/.sbt/0.13/plugins/目录可以认为是一个标准的SBT项目， 这个项目的配置文件中配置的插件即认为是全局的global plugins。

所以我们在~/.sbt/0.13/plugins/plugins.sbt中添加如下配置：

addSbtPlugin("net.virtual-void" % "sbt-dependency-graph" % "0.7.5")

一般的插件，这样就OK了，但是sbt-dependency-graph插件还有点儿设置需要配置，所以，还要在~/.sbt/0.13/global.sbt中添加如下内容：

net.virtualvoid.sbt.graph.Plugin.graphSettings

以上两步都搞定之后，就可以执行如下命令来分析当前项目的依赖关系了：

sbt dependency-graph

2. SBT免翻墙技巧

个人方案配置，在build.sbt里添加resolvers：

resolvers += "Local Maven Repository" at "file://"+Path.userHome.absolutePath+"/.m2/repository"

resolvers += "OSChina Maven Repository" at "http://maven.oschina.net/content/groups/public/"

externalResolvers := Resolver.withDefaultResolvers(resolvers.value, mavenCentral = false)

说明：SBT默认的Resolvers(ivy local和mavenCentral)， 我们希望依赖管理走本地maven库和公司内部库，所以， 通过覆盖原来的externalResolver的值来达到目的， 即上面配置中mavenCentral = false对应行。

2.使用SBT插件生成IDEA项目结构。 插件地址：https://github.com/mpeltonen/sbt-idea
